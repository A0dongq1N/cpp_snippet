# fork 不出新进程了！一个被 99% 的人忽视的容器陷阱

> 没有内存泄漏、没有 CPU 打满，但你的服务就是起不来——凶手可能是那些你看不见的僵尸进程。

---

## 凌晨三点，服务瘫痪

凌晨三点，告警炸了：网关返回 502。你火速登录服务器，发现容器还在跑，内存和 CPU 一切正常，但任何新请求都失败。你尝试进容器排查——

```bash
$ docker exec -it my-service bash
docker: Error response from daemon: 
  OCI runtime exec failed: exec failed: 
  unable to start container process: 
  fork/exec /usr/bin/bash: resource temporarily unavailable
```

连 `exec` 都进不去。你直接看宿主机上的进程列表：

```
root  23  0.0  0.0  0  0 ?  Z   10:00  0:00 [child-1] <defunct>
root  24  0.0  0.0  0  0 ?  Z   10:00  0:00 [child-2] <defunct>
root  25  0.0  0.0  0  0 ?  Z   10:00  0:00 [child-3] <defunct>
...
```

你直接看宿主机上的进程列表：

```
$ ps aux | grep defunct
root  1234  0.0  0.0  0  0 ?  Z   02:00  [child-1] <defunct>
root  1235  0.0  0.0  0  0 ?  Z   02:00  [child-2] <defunct>
...
root 32767  0.0  0.0  0  0 ?  Z   02:50  [child-32700] <defunct>
```

密密麻麻，PID 已经接近 32768 的上限。

> **这就是僵尸进程的杀伤力——它不吃内存不吃 CPU，但它吃 PID。PID 吃完了，系统就无法创建任何新进程。**

这场景在容器环境尤为致命：容器的 PID 1 往往就是你的业务进程，它如果不负责任，谁也救不了。今天我们用一个小 Demo 把来龙去脉拆解清楚。

---

## 一、僵尸进程：一个"不占内存的内存泄漏"

**僵尸不占物理内存，只占 PID。但 PID 是有限资源。**

内核工作原理：

1. 子进程调用 `exit()` 退出，进程实体被回收，**唯独进程表项（task_struct）被保留**；
2. 内核保留这份"死亡证明"，等待父进程调用 `wait()` / `waitpid()` 来读取退出状态；
3. 只有父进程 `wait()` 之后，内核才会彻底释放这个表项，**同时回收 PID**。

所以一个僵尸进程本身的开销微不足道——但系统中 PID 总量有硬上限（默认 32768，`kernel.pid_max`）。当僵尸数量 + 正常进程数逼近上限时：

- `fork()` 返回 `EAGAIN`（Resource temporarily unavailable）
- 容器内任何新命令都无法执行（包括 `docker exec`、健康检查、日志收集）
- 服务假死——活着但不响应，比崩溃更可怕

这本质上就是一个**PID 泄漏**。每个没被 wait 的僵尸，都在永久占用一个 PID 槽位。

一句话总结：

> **子进程退出后，父进程如果不 wait，它就永远是僵尸。**

---

## 二、谁最容易成为"不负责的家长"？

在传统 Linux 环境中，如果父进程先于子进程退出，子进程会自动"过继"给 **PID 1（init）**。而传统的 init 进程（如 SysV init、systemd）会负责任地周期调用 `waitpid(-1, ...)` 收割所有孤儿。

**但在容器里，情况截然不同：**

- 容器的 PID 1 **就是你的 ENTRYPOINT**；
- 如果你的业务进程作为 PID 1 运行，它很可能**根本没有 wait 逻辑**；
- 任何孤儿进程被过继过来后，就永远没人收割了。

这就是容器环境中僵尸进程泛滥的核心原因。

## 三、Demo：亲手制造僵尸，再亲手消灭它

我们准备了一个完整可跑的 Demo，完整代码见文末。先看架构：

```
项目文件结构：
├── zombie_factory.c   # 制造 8 个短命子进程的"僵尸工厂"
├── bad_init.c         # Scene 1 的 PID 1：一个有 bug 的业务进程
├── start.sh           # Scene 2 的入口脚本
├── monitor.sh         # 每 3 秒观测进程树和僵尸数量
├── Dockerfile         # Scene 1 镜像
├── Dockerfile.tini    # Scene 2 镜像（用 tini 作为 PID 1）
└── run_demo.sh        # 一键运行对比实验
```

### 3.1 僵尸工厂：zombie_factory.c

这个程序做了两件事：

1. **fork 出 8 个子进程，它们立刻 exit**——此时子进程变成僵尸，附属于 zombie_factory；
2. **zombie_factory 自己 sleep 4 秒后退出，不 wait 任何子进程**——8 个子进程孤儿化，被过继给 PID 1。

```c
#define NUM_CHILDREN    8
#define PARENT_LIVE_SEC 4

int main(void) {
    for (int i = 0; i < NUM_CHILDREN; i++) {
        pid_t c = fork();
        if (c == 0) {
            // 子进程立即退出
            exit(0);
        }
        // 父进程不管，继续 fork 下一个
    }
    sleep(PARENT_LIVE_SEC);  // 4 秒后退出，孤儿过继给 PID 1
    return 0;                // 注意：全程没有 wait()！
}
```

**时序图：**

```
t=0s  fork child-1..8 → 子进程立即 exit → 成为 zombie_factory 的僵尸
t=4s  zombie_factory 退出 → 8 个子进程变成孤儿 → 过继给 PID 1
```

到这一步，8 个僵尸的命运就完全取决于 **PID 1 是谁**了。

### 3.2 Scene 1：一个不负责的 PID 1（bad_init）

```c
int main(void) {
    // fork 僵尸工厂
    if (fork() == 0) {
        execl("/app/zombie_factory", "zombie_factory", NULL);
    }

    // fork 监控脚本
    if (fork() == 0) {
        execl("/bin/bash", "bash", "/app/monitor.sh", NULL);
    }

    // ⚠️ 这里就是 bug：永不调用 waitpid()
    while (1) sleep(86400);
}
```

`bad_init` 模拟的是一个**真实业务进程**——它 fork 子进程干活，但从未想过要 wait。它作为容器的 PID 1 运行时，zombie_factory 退出后，8 个孤儿归它收养，然后……就没有然后了。

**进程树：**

```
bad_init (PID 1, 不调用 waitpid)
  ├─ zombie_factory → fork 8 子进程后退出
  │    └─ child-1..8 (僵尸，PPID 变为 bad_init)
  └─ monitor.sh
```

**运行效果：**

```
===== 10:00:15 [t=6s] =====
>>> ZOMBIE COUNT: 8 <<<
--- zombie detail ---
root  23  0.0  0.0  0  0 ?  Z   10:00  [child-1] <defunct>
root  24  0.0  0.0  0  0 ?  Z   10:00  [child-2] <defunct>
...

===== 10:00:30 [t=21s] =====
>>> ZOMBIE COUNT: 9 <<<   ← 还在涨！
```

9 个僵尸，一个不少，而且**永远存在**（直到容器被销毁）。

### 3.3 Scene 2：引入 tini，让专业的来做 PID 1

**tini** 是一个极简的 init 进程，核心逻辑就一件事：

```c
// tini 的核心伪代码
while (1) {
    waitpid(-1, &status, WNOHANG);  // 收割所有已退出的子进程
    // 转发信号给子进程...
}
```

在 `Dockerfile.tini` 中：

```dockerfile
ENTRYPOINT ["/usr/bin/tini", "--"]
CMD ["/bin/bash", "/app/start.sh"]
```

tini 作为 PID 1，你的业务逻辑以 `CMD` 方式作为 tini 的子进程运行。zombie_factory 退出后，8 个孤儿归 tini，tini 立刻 `waitpid(-1)` 收割。

**进程树：**

```
tini (PID 1, 循环 waitpid(-1))
  └─ start.sh → exec monitor.sh
       └─ zombie_factory (后台) → fork 8 子进程后退出
            └─ child-1..8 → 孤儿归 tini → 被立即 reap
```

**运行效果：**

```
===== 10:01:15 [t=6s] =====
>>> ZOMBIE COUNT: 0 <<<     ← 干干净净！

===== 10:01:30 [t=21s] =====
>>> ZOMBIE COUNT: 0 <<<     ← 一直是 0
```

同样的 zombie_factory，同样的 8 个子进程——唯一的区别就是 PID 1 换成了 tini。

## 四、monitor.sh：我们的观测窗口

监控脚本每 3 秒打印一次进程树和僵尸数量，核心逻辑极简：

```bash
# 统计所有 status = Z 的进程
zombie_count=$(ps aux | awk '$8=="Z"{c++} END{print c+0}')
echo ">>> ZOMBIE COUNT: ${zombie_count} <<<"

# 如果有僵尸，展示详情
if [ "$zombie_count" -gt 0 ]; then
    ps aux | awk 'NR==1 || $8=="Z"'
fi
```

配合 `ps -ef --forest` 展示进程树，整个过程一目了然。

## 六、为什么会踩这个坑？三个常见场景

### 场景一：直接以业务进程作为 ENTRYPOINT

```dockerfile
# ❌ 危险写法
ENTRYPOINT ["./my-server"]
```

`my-server` 作为 PID 1，如果它内部有子进程管理逻辑但从不 wait，僵尸必然堆积。

### 场景二：Shell 脚本作为 ENTRYPOINT + 后台任务

```dockerfile
# ❌ 危险写法
ENTRYPOINT ["/bin/bash", "/entrypoint.sh"]
```

如果脚本里用了 `some-task &`，子进程退出后僵尸会挂到 bash（PID 1），而 bash 不一定会主动 wait 后台任务。

### 场景三：多进程容器（Supervisor 模式）

在一个容器里跑多个服务（如 Nginx + PHP-FPM），主进程 fork 了多个子进程，某个子进程异常退出后如果不被主进程 wait，就会变成僵尸。

---

## 七、进阶讨论：Shell 脚本 + 带 reap 能力的程序，能避免僵尸吗？

一个常被问到的场景：

```dockerfile
ENTRYPOINT ["/bin/bash", "/start.sh"]
```

`start.sh` 里启动了一个有 wait 能力的程序作为子进程。这个 reaper 能把僵尸都收掉吗？

**答案分三种情况。**

### 情况一：`exec` 到 reaper ✅ 可行

```bash
#!/bin/bash
# start.sh
exec /app/my-reaper --business-server
```

`exec` 会让 reaper 原地替换 shell，**继承 PID 1**。此后所有孤儿都归 reaper，全部可收。

> 本质上这和 `ENTRYPOINT ["/app/my-reaper"]` 没有任何区别——reaper 就是 PID 1。

### 情况二：reaper 作为 shell 的子进程 ❌ 无效

```bash
#!/bin/bash
# start.sh — bash 是 PID 1
/app/my-reaper &        # 后台子进程
/app/business-server    # 前台阻塞
```

进程树：

```
bash (PID 1)
  ├─ my-reaper (带 wait 能力)
  └─ business-server
       └─ short-lived-child → exit → 孤儿归谁？
```

`short-lived-child` 退出后，孤儿会顺着进程树**往上找活的祖先**，一路走到 **bash（PID 1）**，不会横跳到 `my-reaper` 这个"叔叔"身上。

**reaper 只能收割自己的直系子孙，收不到旁系孤儿。**

### 情况三：`PR_SET_CHILD_SUBREAPER` ⚠️ 有条件可行

Linux 3.4+ 支持 **子收割者（subreaper）**：

```c
prctl(PR_SET_CHILD_SUBREAPER, 1, 0, 0, 0);
```

设置后，该进程的**整个后代树**中产生的孤儿，都会优先过继给它，而非 PID 1。

但这里有一个容易被忽略的约束：**subreaper 只收割自己后代的孤儿**。如果 business-server 和 my-reaper 是兄弟关系，business-server 的孤儿照样归 bash。你必须让 reaper 成为 business-server 的**祖先**才行：

```bash
#!/bin/bash
# start.sh
exec /app/my-reaper -- /app/business-server
# my-reaper 现在是 PID 1，下属所有孤儿都归它
```

> 补充：Demo 中的 `bad_init.c` 也调用了 `PR_SET_CHILD_SUBREAPER`，但它本身就是 PID 1，所以这个调用是冗余的——PID 1 天然就是全局 reaper。

### 孤儿归属规则（一句话版）

```
孤儿归谁？
→ 顺着进程树往上，找第一个活着的祖先
→ 如果路上遇到 subreaper，就停在它那里
→ 否则一路走到 PID 1
```

**关键不是"有没有 wait 能力"，而是"在不在孤儿的上溯路径上"。**

## 八、根治方案

| 方案 | 适用场景 | 优缺点 |
|------|---------|--------|
| **tini** | 通用方案，尤其适合单进程容器 | 极小（~50KB），有信号转发，开箱即用 |
| **dumb-init** | 类 tini，一些发行版内置 | 功能类似，看团队偏好 |
| **docker --init** | Docker 原生支持 | `docker run --init` 一行搞定，无需改镜像 |
| **业务代码里 wait** | 有子进程管理的业务 | 最根本的方案，但容易遗漏 |

### 推荐做法

```dockerfile
# ✅ 推荐写法一：镜像内置 tini
RUN apt-get install -y tini
ENTRYPOINT ["/usr/bin/tini", "--"]
CMD ["./my-server"]

# ✅ 推荐写法二：运行时加 --init
docker run --init --rm my-image
```

**额外注意**：tini/dumb-init 这类轻量 init 在信号处理上也更可靠。业务进程直接当 PID 1 时，内核会忽略发给 PID 1 但没有 handler 的信号（这是内核的默认行为），导致 `docker stop` 发 SIGTERM 后进程不能优雅退出，最后被 SIGKILL 强杀。

## 九、总结

这个问题的恐怖之处在于它的隐蔽性：

- **你的监控面板一切正常**——内存、CPU、磁盘 IO 全是绿的；
- **但用户已经炸了**——502 刷屏，健康检查超时；
- **你甚至进不去容器排查**——`docker exec` 直接报 `resource temporarily unavailable`。

整个链条其实很简单：

```
子进程退出 → 父进程不 wait → 僵尸占着 PID 不释放
                ↓
        父进程先退出 → 孤儿归 PID 1
                ↓
     容器 PID 1 是你的业务进程
                ↓
   业务进程从不 wait → PID 持续泄漏 → 最终耗尽 PID 池，系统瘫痪
```

本质就是一个**PID 泄漏**，和内存泄漏一样致命，但监控更难发现。

解决方案同样简单：**确保 PID 1 会 wait 子进程。**

- 开发环境：`docker run --init`
- 生产镜像：`tini` 或 `dumb-init` 作为 ENTRYPOINT
- 自研服务：在代码里主动 `waitpid(-1, WNOHANG)` 或在 Go 里 `reaper.SysProcAttr`

两行 Dockerfile 的改动，能避免凌晨三点的一场生产事故——这笔账不用我帮你算了。

## 附录：完整源码

> 所有代码可在 GitHub（项目地址）找到。以下为核心文件。

### zombie_factory.c

```c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <sys/types.h>

#define NUM_CHILDREN    8
#define PARENT_LIVE_SEC 4

int main(void) {
    printf("[zombie_factory] PID=%d PPID=%d: forking %d children...\n",
           getpid(), getppid(), NUM_CHILDREN);
    fflush(stdout);

    for (int i = 0; i < NUM_CHILDREN; i++) {
        pid_t c = fork();
        if (c < 0) { perror("fork"); exit(1); }
        if (c == 0) {
            printf("[child-%d] PID=%d exiting\n", i+1, getpid());
            fflush(stdout);
            exit(0);
        }
        printf("[zombie_factory] forked child-%d PID=%d\n", i+1, c);
        fflush(stdout);
    }

    printf("[zombie_factory] sleeping %ds then exit (no wait)\n", PARENT_LIVE_SEC);
    fflush(stdout);
    sleep(PARENT_LIVE_SEC);
    printf("[zombie_factory] exiting → orphans go to PID 1\n");
    fflush(stdout);
    return 0;
}
```

### bad_init.c

```c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <sys/types.h>
#include <sys/prctl.h>

int main(void) {
    prctl(PR_SET_CHILD_SUBREAPER, 1, 0, 0, 0);
    printf("[bad_init] PID=%d — I am PID 1, will NOT reap zombies\n", getpid());
    fflush(stdout);

    if (fork() == 0) {
        execl("/app/zombie_factory", "zombie_factory", NULL);
        perror("exec zombie_factory"); exit(1);
    }

    if (fork() == 0) {
        execl("/bin/bash", "bash", "/app/monitor.sh", NULL);
        perror("exec monitor.sh"); exit(1);
    }

    while (1) sleep(86400);
    return 0;
}
```

### Dockerfile（Scene 1：僵尸堆积版）

```dockerfile
FROM ubuntu:22.04
RUN apt-get update && apt-get install -y gcc procps && rm -rf /var/lib/apt/lists/*
WORKDIR /app
COPY bad_init.c zombie_factory.c monitor.sh ./
RUN gcc -Wall -o bad_init bad_init.c \
 && gcc -Wall -o zombie_factory zombie_factory.c \
 && chmod +x monitor.sh
ENTRYPOINT ["/app/bad_init"]
```

### Dockerfile.tini（Scene 2：tini 收割版）

```dockerfile
FROM ubuntu:22.04
RUN apt-get update && apt-get install -y gcc tini procps && rm -rf /var/lib/apt/lists/*
WORKDIR /app
COPY zombie_factory.c start.sh monitor.sh ./

 && chmod +x start.sh monitor.sh
ENTRYPOINT ["/usr/bin/tini", "--"]
CMD ["/bin/bash", "/app/start.sh"]
```

### monitor.sh

```bash
#!/usr/bin/env bash
echo "[monitor.sh] PID=$$, PPID=$PPID — zombie watcher started"

tick=0
while true; do
    sleep 3
    tick=$((tick + 3))
    echo ""
    echo "===== $(date '+%H:%M:%S') [t=${tick}s] ====="
    echo "--- ps --forest ---"
    ps -ef --forest | head -25
    echo ""
    zombie_count=$(ps aux | awk '$8=="Z"{c++} END{print c+0}')
    echo ">>> ZOMBIE COUNT: ${zombie_count} <<<"
    if [ "$zombie_count" -gt 0 ]; then
        echo "--- zombie detail ---"
        ps aux | awk 'NR==1 || $8=="Z"'
    fi
done
```
